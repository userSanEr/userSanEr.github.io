webpack是什么

webpack是一种前端资源的构建工具，一个静态的模块打包器。

在webpack看来，前端的所有资源(js/json/css/img/less/…)都会作为模块处理。

它会根据模块的依赖关系进行静态解析，打包生成对应的静态资源(bundle)

webapck的5个核心

entry

用来指示webpack以哪个文件为入口起点开始打包，分析构建内部依赖图。

output

指示webpack打包后的资源bundles输出到哪个目录下，以及如何命名

loader

让webpack能够去处理自身不能处理的文件

plugins

用于执行范围更广的任务。插件的范围包括，从打包优化和压缩，一直到重新定义环境中的变量等

mode

用来指示webpack使用相应模式的配置

选项 配置 环境
development process.env.NODE_ENV=’development’ 开发环境
production process.env.NODE_ENV=’production’ 生产环境

webpack的初步使用

如果使用webpack构建工具，则需要下载webpack包

  • 初始化package.json文件，输入指令npm init

  • 下载安装webpack，输入指令npm i webpack webpack-cli -g

    生产环境、开发环境可以自己选择npm i webpack -D

    要注意webpack在你下载了全局文件时，在每个文件中还需要单独下载一次

编译打包应用 在我们创建文件之后，运行指令便可以进行打包

  • 开发环境指令 webpack src/js/index.js -o build/js/built.js --mode=development

    功能：webpack自身可以编译打包js和json文件，可以将es6的模块化语法转化成浏览器能识别的语法。

  • 生产环境指令webpack src/js/index.js -o build/js/built.js --mode=production

    功能：在开发环境的功能上，增加压缩代码的功能

说到这里以上是我们没有配置任何webpack的配置文件它所起的作用。

总结：

webpack可以自己打包js以及json文件，可以自动将es6的模块化语法转化，能压缩代码

问题：

不能编译打包css、img等文件，不能将js的es6及以上的语法进行转化，其他非js文件也不能进行转化，eg：字体。

webpack的基本功能实现

所有的配置文件都是在与src同级的webpack.config.js文件中进行配置。

webpack.config.js指示webpack干哪些活，webpack构建工具是基于nodejs平台运行的，模块化采用的是commonjs

当使用webpack命令，默认查找webpack.config.js配置文件

HRM

需要devServer中的hot配置为true

当只有一个模块发生变化时，我们只会打包这一个模块，而其他模块则使用之前的缓存

html 只有一个模板文件 所以我们不需要

css 当使用style-loader会默认开启HMR功能

js 使用module.hot监听是否开启了这个功能，但是只能对非入口文件进行监听

source-map

开启方式：在于entry同级设置：devtool: 'inline-source-map'

提供源代码到构建后代码的映射 (如果构建后的代码出错了，通过映射可以追踪源代码错误)

开启的source-map的模式可以分为一下几种

可以搭配使用各模式：[inline | eval | hidden] nosource [ cheap | cheap-module] source-map

  • (inline | eval) - source-map 都是行内

    错误代码准确信息 源代码的错误位置

    • inline 类似于head中引入style标签
    • eval 类似于在行内样式中写style标签

  • (nosource | hidden) - source-map

    不能追踪源代码的错误，只能提示构建后代码的位置

    • nosource 全部代码都隐藏
    • hidden 只隐藏源代码 会提示构建后代码错误

  • (cheap | cheap-module) - source-map

    错误代码准确信息 源代码的错误位置 但是只能精确到行

    • cheap-module会将loader中的代码错误提示信息也加入

不过在开发环境和生产环境有所不同

开发环境：evel-source-map

生产环境：source-map

tree sharking

作用：会自动去除无用代码

eg：当你在项目中引入了某个组件 但是你只是按需引入它的部分功能 那么它的别的功能是不会被打包进去的

开启条件：ES6模块化 生产环境

可以在package.json中进行配置"sideEffects": false代表所有的代码都没有副作用，可以进行tree sharking

问题：可能会把css / @babel/polyfill 文件不打包进去 因为这些文件都是只引入而没有使用的

如果不想css被tree sharking 则使用sideEffects": ["*.css", "*.less"]在这里*代表任意一个css、less文件

oneOf

我们在module里面配置的每一个loader在生效时，会将每一个文件检查一遍，查看是不是自己匹配的文件，

但是我们其实大部分loader只对一个文件起作用，所以我们将只对一个文件起作用的loader配置放在oneOf这个数组中，这样当它查找到一个匹配的时候，后面的就不会再去查找

babel缓存

开启方法： 在babel-loader的options配置项中开启cacheDirectory: true，与presets同级

但是该方法只有在配置服务器的情况下才可以看到，可以先使用node来模仿服务器

只有在第二次打包构建时才会生效，因为第一次是进行缓存的。

但是该缓存开启之后，当我们服务器配置的缓存时间较长，可能在这段配置时间内，修改的内容都不会生效，此时我们就需要使用hash值，来告诉缓存我已经变了

hash - chunkhash - contenthash

为了可以让用户使用体验更好，我们可以使用强制缓存，这样下次直接从缓存获取内容。

​ 但是如果我们缓存的时间较长，当我们改变代码时，他也不会发生改变，此时我们就需要hash值来标记，当hash值发生变化时，就代表当前的模块发生修改

三者区别：

  • hash

    webpack每次打包都会生成hash值，也就是不管代码有没有发生变化，hash值肯定会变，这样会导致，文件内容并没有发生变化，但是文件名变了，就会重新加载

  • chunkhash

    如果打包来自同一个入口，那就同属于同一个chunk，所以就属于同一个hash值，比如css是从js中引入的，当js发生改变时，css也会跟着改变

  • contenthash

    是根据文件内容去生成hash值，文件内容不同，hash值也不同，不管是不是来自同一个入口

代码分割 code split

主要是进行代码分割

在我们一般不进行代码分割时，会把所有的文件都打包生成一个js文件，但是这样会导致代码体积比较大，打包缓慢

在使用code split之后，就可以使每个js文件分割，然后单独打包，这样所有的js文件就可以并行打包构建

  • 将node_modules单独打包成一个js

  • 使用import动态引入js，然后对该文件进行单独打包

    在这里使用/**/包裹住的部分可以用来对打包生成的js文件进行重命名

懒加载、预加载

懒加载：就是当你需要这个文件的时候你才去加载，当我们使用import方法引入的时候就相等于懒加载

预加载：是在所有的主要文件都下载完之后，浏览器空闲了，悄悄的去加载之后的文件。

配置方法：

webpackPrefetch: true就是对该组件/方法进行预加载

但是预加载的兼容性非常差，只能在一些高版本的浏览器中才可以使用

externals

将一些包在打包时排除出去，然后在模板html中使用script将cdn链接引进来

为什么使用cdn引入的话，打包速度就会变快呢

因为我们使用的是线上的资源，进而不会影响打包速度

注意：需要在html中使用script引入

开启：在于entry同级配置externals

多进程打包

多进程打包 需要下载 thread-loader 一般用在babel-loader上

但是也可以用在任何一个loader上，你用在哪个上它就对哪个起作用

缺陷：进程启动大概为600ms，进程通信也有开销，只有工作消耗时间比较长，才需要进程打包

要想体现出来，就得babel多用几次 babel用得多也就是js文件要多 ，如果比较少的话可能得不偿失

开启配置 在你用在哪个文件里，加上这个loader就行

webpack配置详情

webpack.config.js文件中各个对象的前后顺序是没有区别的

entry 入口起点

entry格式可以有三种写法：string、array、object

这三种写法可以同时使用

  • string

    单入口

    打包形成一个chunk，输出一个bundle文件，此时的chunk名字默认main文件

  • array

    多入口，但是数组方式，所有的入口文件只会形成一个chunk，输出去只有一个bundle文件

    最后还是会将数组的所有入口打包到数组的第一项里面

    eg：

    • 当我们想让html的HRM功能生效时，会将html文件与入口文件同时配置到数组中

    ​ 但是HRM只能在生产环境使用

    • 在使用dll将第三方包单独打包时，我们也可以使用数组的方式

      可以将这三个第三方包打包生成一个文件

  • object

    多入口，有几个文件就有几个chunk，同时输出去就有几个bundle

    此时chunk的名称就是entry对象中key的值

output

在指定目录那里可以设置hash值，这个在缓存中有详细的讲解

非入口chunk的名称 —- 是指我们使用import动态引入的文件打包之后的名字

整个库向外暴露的变量名、以及将变量添加到哪里 —- 是指我们使用dll打包第三方库时，第三方库的名字

module 进行loader配置

loader的配置顺序，为倒叙，先使用的loader写在最后

不同的文件配置不同的loader来处理，每一个对象里只能处理一种情况

eg：我们解析css，使用多个loader，将css-loader写在最后。

  • test：表示配置那些文件，如果没有限制的话，会遍历所有的文件，看见是这个结尾就会去处理

  • use：

    • 使用多个loader则使用use:[],在数组中写入所有使用的loader名称，
      • 单个loader则使用loader:loader名称进行配置

  • options：只是针对某个loader做的设置，在使用时必须用在单独的loader下才能生效

  • exclude： 表示排除哪些文件

  • include：表示只检测哪些文件

  • enforce：’pre’ 优先执行 ‘post’ 延后执行 一般不写的话 按默认顺序执行

resolve 解析模块规则

  • alias: 配置解析路径别名，优点：简写路径，缺点：写路径没有提示

  • extensions： 配置省略文件路径的后缀名，会为没有后缀名的文件依次补充：js、json、css等后缀名，按照配置顺序依次匹配，如果都没有就会报错

  • modules：告诉他去哪个模块里找这个第三方库，我们之前没有使用../标记该目录在哪里的时候，他会一层一层的去查找，直到找到这个node_modules.。

    但是当我们指定了路径之后，他就会直接去查找这个路径，比较快捷，更好的性能

    配置在modules数组里的每一项都会去查找，如果第一项没有，就会自动匹配第二项

devServer

只能用于开发环境

optimization

用在生产环境才会有比较大的作用，主要是对代码打包时的优化配置

  • splitChunks：该配置配置之后，会将node_modules里面的代码进行单独打包，除了chunk这个配置项之外，还有许多配置项，但是一般都使用默认配置不会去更改

  • runtimeChunk：该模块是针对我们使用import引入的模块

    eg：我们在主入口文件中，使用import引入了一个js模块，虽然我们使用import引入，会将该模块进行单独打包，但是当这个模块发生变化时，相对应的hash值，也会发生变化，则连带着主入口文件中，引入该模块的hash模块也发生变化，所以该模块与主入口文件都会重新加载，但是当我们开启这个配置项之后，会将这个引入部分单独作为一个模块打包，当该模块发生变化时，只有该模块与引入部分发生变化，其余不变。

  • minimizer：配置生产环境压缩js、css方案
